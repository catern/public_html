Making it easier to run distributed software

Distributed software, which accesses and depends on remote resources, is harder to run than local-only software. I'm interested in making it easier to run distributed software for both programmers and end-users. I think this will promote the development and use of open source distributed software, which will in turn lead to more high-quality distributed software being available.

  • The Unix process API is unreliable and unsafe
  • For fast protocols, be backwards-compatible in the client, not on the wire
  • Write libraries instead of services, where possible
  • For me, the best number of computers is one

Taking lambda-abstraction as far as it can go

I'm interested in doing practical programming with just lambda-abstraction and application. Many features, both at the level of surface syntax (such as pattern matching, DSLs) and type system (such as effect systems, access controls, DSLs) are easily implementable with just lambda-abstraction and application, and this allows them to be used productively in languages without such advanced features built-in. I'm interested in taking this further, because I think this helps make such innovations more widely usable.
  • Write code, not configuration
  • Write code, not compilers
  • Write signatures, not schemas
  • Write modules, not microkernels
  • Cursed PL Iceberg
  • Concurrency based on shift/reset and object-oriented effect handlers
  • Inheritance was invented as a performance hack
  • public/protected/private is an unnecessary feature
